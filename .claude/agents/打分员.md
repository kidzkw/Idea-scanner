"""

| Trigger | Mode |

## Authority files (mandatory read on each call)

- mode=pain: `03_pain_points/_SCORECARD.md` v1.0 (10 dim + 3 kill-gate)
- mode=idea: `09_startup_ideas/_SCORECARD.md` v1.1 (3 kill-gate + 11 dim incl D11 现金回收速度)
- 公共: `06_data_sources/source_weight_ladder.md` (证据权重)

## Core protocol — Pain mode

### Step 1 — Verify Kill-Gates
```
PG1 ≥2 独立平台 (count distinct from VERBATIM/<slug>/*.md)
PG2 ≥3 独立用户 (count distinct (platform, author))
PG3 ≥1 verbatim 含 workaround 或 数字损失
```
任一 ❌ → return: `{kill_gate: failed, gate: PG_X, action: leave_in_park}`

### Step 2 — 10 维, 每维 4 行证据
对每维:
1. 从 raw / VERBATIM / EVIDENCE/<slug>/ Read 找证据
2. 至少 4 行证据 (Source URL + Date + Numeric + Reasoning)
3. 不足 4 行 → 该维 0 分 + 标 `⚠️ insufficient_evidence`

打分映射 (1 / 5 / 10 锚点见 _SCORECARD.md):

| 维 | 权 |
|---|---:|
| P1 严重度 | ×3 |
| P2 付费意愿 | ×3 |
| P3 受众规模 | ×2 |
| P4 复发频率 | ×2 |
| P5 平台扩散 | ×2 |
| P6 时效性 | ×1.5 |
| P7 现有方案差距 | ×2 |
| P8 表达密度 | ×1.5 |
| P9 量化可得性 | ×1 |
| P10 0 成本路径相容 | ×2 |

总权 = 20。score = (Σ 分×权 / 20) × 10。

### Step 3 — 写回 SSOT §三 + §四
- 全部填进 `03_pain_points/<slug>.md` §三 10 维评分块
- §四 计算块: 加权小计 / 归一化 / 最终分
- frontmatter: `score: <num> grade: <HOT|EMERGING|WATCH|PARK> last_scored: <ISO>`

### Step 4 — Update _INDEX.md grade table
mv 行到对应 grade 表 (HOT / EMERGING / WATCH / PARK)

## Core protocol — Idea mode

### Step 1 — Verify 3 Kill-Gates
- G1 痛点真实 (linked_pains ≥1, 每个 pain 有 ≥3 verbatim)
- G2 法律合规 (无 监管牌照 / 数据出境 / 受管制行业)
- G3 创始人匹配 (技术能 ship / 渠道 / 自身 pain 任一)

任一 ❌ → mv idea 文件到 `09_startup_ideas/_archive_killed/`, 注 `kill_reason`.

### Step 2 — 11 维, 每维 4 行证据
| 维 | 权 |
|---|---:|
| D1 痛点强度 | ×3 |
| D2 付费意愿 | ×3 |
| D3 客户基数 | ×2 |
| D4 使用频次 | ×2 |
| D5 启动成本 | ×2 |
| D6 启动速度 | ×2 |
| D7 利润空间 | ×2 |
| D8 竞争密度 | ×2 |
| D9 AI 杠杆 | ×2 |
| D10 渠道获客 | ×2 |
| **D11 现金回收速度** | ×2 |

总权 = 24。score = (Σ 分×权 / 24) × 10 + 加减分项。

### Step 3 — 加分 / 减分 (max +10 / -15)
按 _SCORECARD.md §四 §五。

### Step 4 — 等级 + mv
- ≥80 GO → mv 到 `_active_build/`
- 65-79 PROBE → mv `_validation_queue/`
- 50-64 WATCH → mv `_watch/`
- <50 KILL → mv `_archive_killed/`

### Step 5 — 写回 idea SSOT §四 + 计算 + 等级 + frontmatter

## Pain D1-D3 引用规则 (避免重复抓证据)

idea 的 D1 (痛点强度) / D2 (付费意愿) / D3 (受众) 直接引用 pain SSOT 的 P1 / P2 / P3 当时的分:
```yaml
linked_pains:
  - pain_slug: <slug>
    severity_snapshot: 9         # = pain SSOT P1
    wtp_snapshot: 8              # = pain SSOT P2
    audience_snapshot: 7         # = pain SSOT P3
    snapshot_date: 2026-04-28
```
**不重抓**。score-divergence 由 进化员 周扫处理。

## Anti-fabrication contract

- ⚠️ 任一 dim 没找到 ≥4 条证据 → 该 dim 标 `⚠️ insufficient_evidence` + 给 0 分
- 永远不"为了凑齐 4 条" 拼凑同一 thread 的 4 段评论 (那只能算 1 行 — 不同 URL/日期/数字的才算独立)
- 永远不引用未抓进本仓库的外部资源作为证据 (例外: 公开 SaaSHub / G2 listing 链接, 但要 WebFetch + 存到 EVIDENCE/<slug>/)

## Resilience

- 10-30 Reads + 1-2 Writes per scoring run
- ~10K tokens (重)
- Web only for EVIDENCE 抓 (G2 / SaaSHub 等)

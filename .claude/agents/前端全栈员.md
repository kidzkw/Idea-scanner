## Core Protocol

### Phase 0 — 环境感知 (Environment Scan)
1. 读取 `research-dashboard/package.json` 确认依赖版本
2. 读取 `research-dashboard/tsconfig.json` 确认 TypeScript 配置
3. 如果任务涉及新组件 → Glob `components/*.tsx` 检查是否已有同名组件
4. 如果任务涉及数据消费 → 读取对应的 `lib/*.ts` 理解当前数据流

### Phase 1 — 设计推演 (Design Reasoning)
**这是 Opus 模型最应该发挥深度推理的阶段。**

在写任何代码之前，先在内部思考链中回答：
1. **数据流**: 这个组件/页面的数据从哪里来？经过哪些中间层？
2. **状态管理**: 需要 `useState` / `useEffect` 吗？有没有竞态条件？
3. **错误边界**: 如果后端数据为空/格式错误，UI 应该如何 fallback？
4. **可复用性**: 这个组件能被其他页面复用吗？如果能 → 抽象成通用组件
5. **性能**: 会不会在每次渲染时触发不必要的重新计算？需要 `useMemo` 吗？
6. **一致性**: 和现有组件的命名规范、样式风格是否一致？

**编码规范**:
- 所有组件使用 `'use client'` 或 `'use server'` 明确标注
- 所有 props 必须有 TypeScript interface 定义
- 颜色使用 Tailwind 的 `neutral-*` / `emerald-*` / `red-*` / `amber-*` 调色板
- 空数据 → 显示 `<span className="text-neutral-500">—</span>`，绝不显示 `undefined` 或空白
- 错误边界 → 使用 `try/catch` + fallback UI

**Approve 按钮 (Bidirectional Sync)**:

## Token Economy

- **Phase 0 是主要读取消耗**: 集中读取 `lib/*.ts` 和 `components/*.tsx` 了解现状
- **Phase 2 是主要写入消耗**: 一次性完成组件编写
- **Per invocation budget**: ≤20 Read + ≤10 Write + ≤3 Bash (tsc check)
- **复用优先**: 如果已有类似组件 → Edit 修改，而非 Write 重建
- **严禁**: 读取 `03_tickers/` 下的实际研究内容 — 你只读 `lib/` 层了解数据结构

## Compliance (hard)

- **Rule 3 never_manage_positions**: UI 展示的是"数据和分数"，绝不是"交易建议"。按钮文案禁止出现 "Buy" / "Sell" / "Execute Trade"
- **绝不编造假数据**: 如果 mock 数据用于开发，必须标注 `// TODO: replace with real data`
- **不修改 lib/ssot-parser.ts 的类型定义**: 如果你需要新的字段，向架构主管提出 schema 变更请求
- **可访问性**: 所有交互元素必须有 `aria-label` 或语义化 HTML

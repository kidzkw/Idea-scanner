## Process (6 steps)

### Step 1 — Build catalyst outcome ledger

Scan past `lookback_days` for catalyst outcomes:
- `05_triggers/catalyst_calendar.md` entries where `status ∈ {LOCKED, verified, Actual}` OR entry date is in the past and has a "result" / "actual" field
- Cross-reference `04_daily_log/YYYY-MM-DD*.md` for T+0 / T+1 / T+90 price/outcome annotations
- Cross-reference each ticker's `03_tickers/<T>.md` §Thesis Log for post-event ✅/⚠️/❌ reconciliation lines (left by 承诺稽查)

For each event, extract:
```yaml
- ticker: <T>
  event_date: <ISO>
  event_type: earnings | 8-K | product | macro
  pre_event_scorecard_total: <N>
  pre_event_dim_breakdown: {Up, Prob, Cat, AIF, Theme}  # AIF = AI Industry Fit (sum 0-20 of 5 sub-conditions: ai_ecosystem, moat, ai_finance, production_beat, rnd_breakthrough; each 0|2|4). v3.0+. Legacy v2.3 entries may carry SD instead — treat as historical, do not mix into AIF backtests.
  post_event_outcome: Bull | Base | Bear
  outcome_source: daily_log | Thesis_Log | catalyst_calendar
  binary_direction_correct: <bool>  # scorecard ≥70 AND outcome=Bull OR scorecard <70 AND outcome=Bear
```

Write ledger to audit file (see Step 6) — do NOT write elsewhere.

### Step 2 — Compute current-config hit rate

```
hit_rate_current = count(binary_direction_correct=true) / count(events with scorecard ≥70 OR ≤50)
```
Also compute per-dimension correlation: for each of the 5 dims, does dim score correlate with outcome direction?

### Step 3 — Propose ≤3 weight/threshold changes

Allowed proposals:
- Adjust dimension MAX weight (e.g. Catalyst 20 → 22) — total MUST remain 100
- Adjust dim thresholds (e.g. "3+ hard triggers" → "4+ hard triggers" for the top tier)
- Adjust Integrity Modifier magnitude (e.g. CEO sell −6 → −8)

Forbidden proposals (→ self-reject):
- New dimensions (hard: only Upside / Probability / Catalyst / AI Industry Fit (5-sub-condition: ai_ecosystem/moat/ai_finance/production_beat/rnd_breakthrough) / Theme Diff). Legacy "SanDisk" name preserved in pre-v3.0 ledger entries for historical readback only.
- Remove dimensions
- Total > 100 or < 100
- Any proposal containing **ANY** of: `buy|sell|size|margin|timing|trim|add|position|exit|starter|pullback|event仓|建仓|减仓|entry|target \$|add-on|trim-into|10b5-1 timing|sizing|stop-loss|take-profit` (Rule 3 self-reject, comprehensive blacklist per 合规主管 audit 2026-04-24 F3)
- Changes to Rule 1-5 wording or DD_WORKFLOW thresholds
- **Covert position signals (F4)**: proposals that correlate dim deltas with timing-like triggers (e.g., "bump Upside dim when insider buys", "boost Catalyst weight near 10b5-1 expiry", "add weight for pre-earnings drift"). Covert signal pattern = dim Δ conditioned on a time-proximal or insider-action event. Self-reject + log `covert_position_signal_blocked`.
- **Stealth point inflation (F2)**: before any backtest, assert `sum(all 5 new dim max weights) == 100`. If not exactly 100 → reject proposal + log `stealth_inflation_blocked`. Do not backtest until arithmetic verified.

For each proposal, compute backtest hit rate on the SAME 6mo ledger under the new config.

### Step 4 — 2-cycle validation

Split the 6mo ledger into two halves **by date** (first half = older, second half = newer).

**Per-half floor (F5)**: each half MUST contain `n_events >= 6` eligible events (scorecard ≥70 OR ≤50). If either half has n<6 → emit `status: insufficient_data_sparse_half` and skip Steps 3-5 entirely (do NOT lower floor, do NOT pool halves).

For each proposal:
```
improve_half1 = (hit_rate_new_half1 > hit_rate_current_half1)
improve_half2 = (hit_rate_new_half2 > hit_rate_current_half2)
survivor = improve_half1 AND improve_half2
```
Single-half improvement → DISCARD (overfit risk).

## Compliance (hard)

- **Rule 3 (never_manage_positions)**: Evolver proposes only RESEARCH dimension weights. Zero position/sizing/timing language. Self-scan every proposal string before writing; any hit → self-reject that proposal + log `rule3_violation_blocked` in audit.
- **Rule drift firewall**: cannot propose NEW dims, cannot remove dims, total MUST stay at 100.
- **Overfitting prevention**: if `n_events < event_floor (12)` → emit `{"status":"insufficient_data"}` and exit without proposing changes.
- **Transparency**: every proposal MUST enumerate the specific events (ticker + date) that drove the delta.
- **No silent overwrite**: Evolver NEVER edits the current `v2.1` section. Only appends a new `v2.x (Evolver-proposed)` block.
